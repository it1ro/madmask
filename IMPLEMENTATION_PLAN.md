## План итераций и подзадач для разработки проекта в Docker

### Итерация 0: Подготовка окружения Docker

**Цель:** Настроить Docker-среду для разработки Rails 8 приложения, чтобы вся работа велась в контейнерах с самого начала.

**Подзадачи:**
1. Создать `Dockerfile` для разработки на основе Ruby 3.2-slim, установить зависимости для SQLite, Node.js (только если нужно, но для Tailwind через gem не требуется) и Bundler.
2. Создать `docker-compose.yml` с сервисом `web`, монтирующим текущую директорию в `/app`, пробросить порт 3000.
3. Настроить переменные окружения для разработки (например, `RAILS_ENV=development`).
4. Создать `entrypoint.sh` для подготовки базы данных (создание, миграции) при старте контейнера.
5. Запустить контейнер и убедиться, что внутри доступен Ruby и Rails.

---

### Итерация 1: Создание Rails-приложения внутри Docker

**Цель:** Сгенерировать новое Rails 8 приложение с нужными опциями, используя Docker-контейнер.

**Подзадачи:**
1. Войти в контейнер через `docker-compose run web bash` и выполнить команду создания приложения:  
   `rails new . -d sqlite3 -c tailwind --skip-jbuilder`  
   (точка означает текущую директорию, которая смонтирована в `/app`).
2. Настроить `config/database.yml` для SQLite (оставить по умолчанию).
3. Добавить в `Gemfile` необходимые гемы (если чего-то не хватает), запустить `bundle install`.
4. Убедиться, что приложение запускается: `rails server -b 0.0.0.0` и открывается в браузере.
5. Зафиксировать изменения в репозитории.

### Итерация 2: Настройка Hotwire и Tailwind

**Цель:** Проверить и при необходимости донастроить Hotwire, Tailwind, убедиться, что всё работает в Docker.

**Подзадачи:**
1. Убедиться, что в `Gemfile` есть `hotwire-rails`, `turbo-rails`, `stimulus-rails` (в Rails 8 по умолчанию).
2. Запустить `bin/rails hotwire:install` (если не выполнен автоматически).
3. Настроить Tailwind: убедиться, что `tailwindcss-rails` установлен и работает. Проверить, что `app/assets/stylesheets/application.tailwind.css` существует.
4. Запустить сборку Tailwind в фоне: `bin/rails tailwindcss:watch` (или через Procfile).
5. Добавить в `docker-compose.yml` второй сервис для `tailwindcss-watch`, чтобы не блокировать основной контейнер.
6. Проверить, что приложение открывается со стилями Tailwind.
7. Добавить Makefile с релевантными для проекта командами.

### Итерация 2b: Главная страница (лендинг)

**Цель:** Создать атмосферную главную страницу, соответствующую тематике магазина (fantasy, horror, sci‑fi, cyberpunk) и [DESIGN_GUIDELINES.md](DESIGN_GUIDELINES.md). Страница должна задавать tone-of-voice, вести пользователя в каталог и **не подключать Three.js / WebGL** (тяжёлые скрипты загружаются только на странице товара, см. OVERVIEW.md).

**Подзадачи:**

1. **Маршрутизация и контроллер**
   - Создать `PagesController` с методом `home`.
   - Настроить корневой маршрут: `root "pages#home"`.

2. **Layout и общая структура**
   - Шапка (header): логотип/название, ссылка «Каталог» (ведущая на `products_path`, даже если он временно заглушен), возможно место для будущей админки.
   - Подвал (footer): минимальная информация, ссылки на соцсети (иконки), копирайт.
   - Использовать CSS-переменные из дизайн-гайдлайна для фона, текста, акцентов.

3. **Hero-блок**
   - Крупный заголовок (шрифт Cinzel) и подзаголовок (Orbitron) с описанием коллекции и 3D-просмотра.
   - Основной CTA — кнопка «В каталог» (стилизована под тему).
   - **Фон:** градиент (тёмно-синий → фиолетовый → чёрный) + наложение CSS-шума и тонкой кибер-сетки. Анимация градиента (плавное смещение) — опционально.

4. **Декоративный элемент (без WebGL)**
   - SVG-силуэт маски/артефакта с эффектом свечения и/или лёгким parallax при движении мыши (Stimulus-контроллер).
   - Альтернатива: небольшой CSS- или canvas-тизер (без загрузки GLB), например, вращающийся куб с намёком на текстуру, чтобы показать, что 3D — часть концепции.

5. **Блок категорий**
   - 4 чипа/карточки для жанров: fantasy, horror, sci‑fi, cyberpunk.
   - Каждая имеет иконку (SVG), название и цвет, соответствующий токену категории из дизайн-гайдлайна.
   - При ховере — внешнее свечение (box-shadow), лёгкая анимация.
   - На данном этапе ссылки могут вести на `#` или на каталог с предустановленным фильтром (после появления фильтрации в итерации 4).

6. **Секция «Как это работает»**
   - 2–3 шага: «Выбери маску», «Вращай в 3D», «Воплоти образ».
   - Каждый шаг — карточка с иконкой и кратким текстом.
   - При скролле в зону видимости карточки появляются с анимацией fade-in-up (Intersection Observer или CSS-анимация с задержкой).

7. **Социальное доказательство / бренд**
   - Короткий текст о ручной работе, жанрах, уникальности коллекции.
   - Опционально: цитата или слоган с эффектом «печатной машинки» (Stimulus).

8. **Адаптивность и доступность**
   - Полная адаптивность под мобильные устройства (Tailwind-утилиты).
   - Контраст текста не ниже WCAG AA.
   - Все интерактивные элементы имеют `focus`-стили.
   - Учёт `prefers-reduced-motion`: отключение или упрощение анимаций для пользователей с чувствительностью к движению.

9. **Производительность**
   - Только CSS-анимации и лёгкие JS-эффекты.
   - Three.js на главной не подключается (проверить через `content_for` или условную загрузку).
   - Изображения (если используются) оптимизированы.

10. **Связь с будущими итерациями**
    - После появления модели `Product` (итерация 3) можно будет добавить на главную секцию «Новинки» или «Рекомендуемые» (запрос к БД), но в рамках 2b это не требуется.

### Итерация 3: Модель Product и CRUD

**Цель:** Реализовать управление товарами (админ-интерфейс) с использованием Hotwire.

**Подзадачи:**
1. Сгенерировать модель `Product` с полями: `name:string`, `description:text`, `price:decimal`, `category:string`; 3D-модель — `has_one_attached :model_file` (не колонка URL); обложка — `has_one_attached :cover_image`.
2. Создать миграцию и применить.
3. Сгенерировать `ProductsController` с RESTful-действиями, используя `scaffold` для быстрого старта, но адаптировать под Hotwire:
   - Вьюхи должны использовать Turbo Frames для редактирования.
   - Для создания/удаления использовать Turbo Streams.
4. Настроить маршруты: `resources :products`.
5. Добавить простую админ-панель (можно ограничиться только стандартными вьюхами с авторизацией позже).
6. Протестировать CRUD в браузере.

### Итерация 5: Stimulus-контроллер WebGL

**Цель:** Реализовать полноценный 3D-просмотр товара с загрузкой GLB/GLTF **только из Active Storage** (`has_one_attached :model_file`). В Stimulus передаётся URL блоба (`Product#effective_model_url` → `data-webgl-preview-model-url-value`).

**Подзадачи:**
1. Сгенерировать Stimulus-контроллер: `bin/rails generate stimulus webgl_preview`.
2. В контроллере `webgl_preview_controller.js` реализовать:
   - `connect()`: инициализация сцены, камеры, освещения, рендерера.
   - `loadModel(url)`: использование `GLTFLoader` для загрузки GLB/GLTF по URL блоба Active Storage.
   - Добавить `OrbitControls`.
   - Обработка ошибок: при неудачной загрузке отображать вращающийся куб.
   - Адаптация размера canvas к контейнеру (ResizeObserver или событие resize).
   - Очистка при `disconnect()`.
3. Подключить контроллер к контейнеру через `data-controller`; значение URL — из `effective_model_url`.
4. В админке: загрузка файла модели; при необходимости — удаление вложения.
5. Протестировать: файл в storage; отсутствие файла — куб-заглушка.

### Итерация 8: Docker для продакшена и Kamal

**Цель:** Подготовить Dockerfile для продакшена, настроить Kamal для деплоя на VPS.

**Подзадачи:**
1. Создать продакшен-`Dockerfile` (многоступенчатый, с предкомпиляцией assets).
2. Настроить `.dockerignore`.
3. Установить Kamal: `gem install kamal`.
4. Выполнить `kamal init` для создания конфигурации.
5. Отредактировать `config/deploy.yml`:
   - Указать сервер (IP, user, ключ).
   - Настроить volumes: `- "/rails/storage:/rails/storage"` для SQLite.
   - Прописать переменные окружения (RAILS_MASTER_KEY, DATABASE_URL при необходимости).
   - Настроить registry (Docker Hub или GitHub Container Registry).
6. Подготовить production-окружение на сервере: установить Docker, настроить SSH.
7. Выполнить `kamal setup` для первичной настройки.
8. Выполнить `kamal deploy` и проверить, что приложение работает.

### Итерация 8.5: Security hardening + аудит (обязательно перед продом)

**Цель:** Закрыть критичные риски безопасности/поддерживаемости перед реальным деплоем: убрать секреты из репозитория, ввести авторизацию админки, включить CSP и базовые защитные настройки, зафиксировать улучшения производительности и привести документацию к факту.

**Подзадачи:**
1. **Секреты и ключи (критично)**
   - Удалить из git и исключить через `.gitignore`: `config/master.key`, `.envrc` (и любые `.env*` с секретами).
   - **Ротация**: сгенерировать новый `RAILS_MASTER_KEY`, перешифровать `config/credentials.yml.enc`, обновить секреты в CI/Kamal.
   - Отозвать/пересоздать `KAMAL_REGISTRY_PASSWORD` (GHCR token/PAT), обновить GitHub Secrets/Environment.
   - (Опционально, отдельной операцией) Очистка истории репозитория от утёкших секретов (BFG / git filter-repo) + принудительное обновление у всех клонов.

2. **Админка: аутентификация/авторизация (Devise)**
   - Подключить Devise (или эквивалент) и закрыть `namespace :admin` через `before_action :authenticate_user!` в `Admin::ApplicationController`.
   - Добавить роль/флаг администратора и проверку доступа к админ-действиям.
   - Добавить минимальные тесты доступа (неавторизованный → редирект/401; админ → доступ).

3. **CSP и базовые security headers**
   - Включить `config/initializers/content_security_policy.rb` и настроить политику под importmap + Google Fonts + ассеты (и при необходимости HDR в `/public/hdr/*`).
   - Настроить `config.hosts` / `host_authorization` в `production.rb` под `madmask.ilmir.tech` (и исключение для `/up`, если нужно).
   - Проверить `filter_parameter_logging.rb` (и добавить специфичные поля, если появятся: `authorization`, `cookie`, и т.п.).

4. **Безопасность зависимостей и CI**
   - Убедиться, что в CI выполняются `brakeman`, `bundler-audit`, `bin/importmap audit`, `rubocop`, тесты.
   - Настроить политику игнора CVE в `config/bundler-audit.yml` только при обосновании и сроках пересмотра.

5. **Производительность и масштабируемость**
   - Добавить индексы для `products` (минимум `category`, `created_at`; при необходимости составной).
   - Проверить `products#index`/`#show` на N+1 и тяжёлые операции во view (особенно Active Storage variants) и при необходимости вынести подготовку в helper/service.
   - Подтвердить настройки SQLite для production (WAL, volume, бэкапы) с учётом Solid Cache/Queue/Cable отдельных файлов БД.

6. **Документация и статус итераций**
   - Обновить `IMPLEMENTATION_PLAN.md`: убрать устаревшие пункты (например, Three.js «через CDN», если по факту importmap+vendor), добавить эту итерацию как обязательную.
   - Сверить и привести в соответствие `implementation_status/*` с фактическими файлами в репозитории (без противоречий).
   - Добавить пояснение, что `config/cache.yml`, `config/queue.yml`, `config/recurring.yml` — стандартные конфиги Solid* (Rails 8), а не «странные» файлы.

### Примечания к плану

- **Docker с самого начала:** Все команды разработки выполняются внутри контейнеров. Для этого используется `docker-compose up` с тремя сервисами: `web` (Rails), `tailwind` (сборка CSS) и возможно `litestream` для бэкапов.
- **Использование Cursor AI:** План разбит на небольшие итерации, каждую из которых можно поручать AI с указанием контекста (файлы, которые нужно изменить).
- **Гибкость:** При необходимости порядок итераций можно менять. Например, дизайн можно начать раньше, если есть готовые макеты.

Этот план позволит поэтапно создать проект, минимизируя риски и обеспечивая постоянную работоспособность после каждой итерации.

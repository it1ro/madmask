# Сводка нагрузочных замеров (по сохранённым прогонам)

Источник сырых логов: [`load_test_measured_2026-04-03.md`](load_test_measured_2026-04-03.md).

**Условия:** локально, Docker `web`, `RAILS_ENV=development`, `ab -k`, `http://127.0.0.1:3000`. Не продакшен (нет Thruster, Solid Cache/Queue как в Kamal), поэтому цифры — **ориентир по порядку величины**, не SLA.

## Apache Bench (полные прогоны)

| Сценарий | c | n | RPS | Среднее время на запрос (аб) | Failed | Примечание |
|----------|---|---|-----|------------------------------|--------|------------|
| `GET /ru/` | 10 | 1000 | **6.17** | ~1621 ms (по пользователю), ~162 ms «на конкурентное соединение» | 0 | HTML ~62 KB |
| `GET /ru/products` | 10 | 1000 | **4.79** | ~2086 ms / ~209 ms | 0 | HTML ~88 KB |
| `GET /ru/products/1` | 10 | 1000 | **5.01** | ~1997 ms / ~200 ms | 0 | хвост latency до **~20 s** у части запросов (98% < ~10.5 s) |
| `GET /ru/products` | 30 | 800 | **2.99** | ~10.0 s / ~335 ms | 0 | сильная деградация: медиана **~7.7 s**, max **~41.8 s** |

---

## Незавершённые / обрезанные прогоны

- **`GET /ru/products` — n=5000 c=30:** в логе зафиксировано только **1500** из 5000 запросов (сессия прервана); итоговых RPS нет.
- **`GET /ru/` — n=400 c=50:** в файле обрыв на «Completed 100 requests» — **результат не зафиксирован**.

Имеет смысл при необходимости повторить эти два сценария с меньшим `n` или в фоне без лимита времени.

---

## Выводы только по имеющимся данным

1. **Пропускная способность:** при **c=10** и keep-alive приложение устойчиво держит порядка **5–6 RPS** на тяжёлых HTML-страницах (главная, каталог) и **~5 RPS** на карточке товара — при **нулевых failed** в завершённых тестах.
2. **Задержки:** под нагрузкой среднее время ответа на «пользователя» (первая строка *Time per request*) — **порядка 1.6–2.1 s** при c=10; у каталога при **c=30** среднее и хвосты резко растут (медиана ~7.7 s, max > 40 s).
3. **Узкое место по latency:** карточка товара при c=10 даёт **длинный хвост** (до ~20 s у самого медленного запроса при 1000 попытках) — имеет смысл отдельно смотреть запросы к БД/Active Storage/WebGL при профилировании.
4. **Прирост от последних оптимизаций кода** этими прогонами **не измерялся** (нет контрольного «до/после» на одном коммите).

Для сопоставления с продакшеном повторите замеры в образе с `RAILS_ENV=production`, Thruster и `LOAD_TEST_ALLOW_HTTP=true` по инструкции в [`load_testing_ab.md`](load_testing_ab.md).

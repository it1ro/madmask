# Backlog: производительность на 1–2 CPU

Источник: план [`performance_plan_1_2_cpu.md`](./performance_plan_1_2_cpu.md). **Пункт 1 (Puma и CPU)** в беклог не включён — отмечен как сделанный в самом плане.

Контекст: Rails 8, Puma, SQLite + Solid Cache/Queue/Cable, Kamal. Приоритеты ниже следуют исходному документу.

---

## 2. SQLite на малых CPU (высокий приоритет)

- [ ] Не занижать `SQLITE_BUSY_TIMEOUT_MS` на проде при конкурентных записях (единый `busy_timeout` с инициализатором WAL).
- [ ] Пакетные джобы и массовые апдейты — с **низким параллелизмом** в очереди, чтобы не забивать писателей SQLite.
- [ ] Проверить `EXPLAIN` по горячим путям (каталог, карточка, админка); избегать лишних `COUNT` (частично закрыто кэшем счётчика в каталоге).
- [ ] **Резерв:** при росте записей и очередей — отдельный план миграции primary на **PostgreSQL**.

---

## 3. Solid Queue и фон (средний–высокий приоритет)

- [ ] Вынести обработку джоб из Puma: `SOLID_QUEUE_IN_PUMA: false`, роль **`job`** в Kamal с `bin/jobs` (второй контейнер на хосте или отдельная машина).
- [ ] Снизить/настроить `JOB_CONCURRENCY` и потоки в `config/queue.yml` (ориентир: 1 процесс, 2–3 потока на 1–2 CPU; расти только по метрикам очереди).
- [ ] Следить за recurring-уборкой: finished jobs не должны копиться без причины.

---

## 4. Кэширование и отдача статики (средний приоритет)

- [ ] Настроить политику Solid Cache: `max_size` / `max_age` в `config/cache.yml`; мониторить рост `production_cache.sqlite3`.
- [ ] Фрагментный кэш для дорогих partial’ов на главной/каталоге (допустимо устаревание на минуты).
- [ ] `Cache-Control` / `ETag` для публичных GET без персонализации (осторожно с сессией и Turbo).
- [ ] CDN / прокси: статика и медиа (Cloudflare или nginx `expires`) — разгрузить Ruby.

---

## 5. Приложение и виды (средний приоритет)

- [ ] Active Storage: не генерировать тяжёлые варианты синхронно в запросе; готовые заранее или lazy через джобы.
- [ ] Убрать N+1: `includes` / `preload` на списках; проверить карточку товара (галерея, вложения).
- [ ] Профилирование страницы товара (`rack-mini-profiler`, логи SQL) — длинный хвост latency по замерам.
- [ ] При росте cookie — рассмотреть `session_store :cache_store` (меньше повторной работы).

---

## 6. Thruster и логирование (низкий–средний приоритет)

- [ ] Убедиться, что Thruster реально используется в прод-образе Kamal (сжатие ответов).
- [ ] Под нагрузкой: `RAILS_LOG_LEVEL=warn` для снижения I/O логов.
- [ ] Не добавлять тяжёлые проверки в `/up` (healthcheck остаётся лёгким).

---

## 7. Измерения (обязательный контур)

- [ ] После значимых изменений перфа повторять `bin/load_test_ab` и `bin/load_test_latencies` на одном стенде (см. `load_testing_ab.md`).
- [ ] Следить за размерами `storage/*.sqlite3` и очередью Solid Queue на корреляцию с деградацией.
- [ ] На хосте с 2 CPU под нагрузкой — краткий профиль CPU (`perf`, `rbspy` или `top`): Ruby vs SQLite vs воркеры.

---

## 8. Рекомендуемый порядок внедрения (дорожная карта)

1. [ ] Зафиксировать baseline (RPS, p95, размеры БД) в текущей конфигурации Kamal.
2. [ ] Очередь отдельно от Puma или снижение `JOB_CONCURRENCY`, если джобы заметны.
3. [ ] Точечно: фрагментный кэш + CDN/статика.
4. [ ] Профилирование карточки товара и тяжёлых SQL.
5. [ ] Пересмотр потоков Puma и при необходимости 2 воркера только при доказанной выгоде на 2 CPU.
6. [ ] Долгосрочно при росте нагрузки на запись — PostgreSQL.

# План мероприятий: производительность на сервере с 1–2 CPU

Контекст: madmask (Rails 8, Puma, SQLite + Solid Cache/Queue/Cable, Thruster в проде, Kamal). На **1–2 ядрах** узкие места — **GVL в Ruby**, **конкуренция за SQLite** и **совместное использование CPU вебом и фоновыми задачами**.

Ниже — приоритетный перечень мероприятий без обязательного перехода на PostgreSQL; порядок отражает типичный эффект / сложность.

**Беклог (пункты 2–8, чеклисты):** [`performance_backlog_1_2_cpu.md`](./performance_backlog_1_2_cpu.md).

---

## 1. Puma и распределение CPU (высокий приоритет)

| Мероприятие | Для 1 CPU | Для 2 CPU |
|-------------|-----------|-----------|
| **`WEB_CONCURRENCY`** | Оставить **1 воркер** (по умолчанию), не раздувать число процессов без замеров. | Опционально **2 воркера** только если RPS упирается в CPU и профилирование показывает пользу; иначе снова **1 воркер + больше потоков**. |
| **`RAILS_MAX_THREADS`** | Держать **5–8** для IO-bound (ожидание БД/сети); выше — чаще хуже latency из‑за GVL. | То же; выровнять с `max_connections` в [`config/database.yml`](../config/database.yml). |
| **Solid Queue в Puma** | При заметной нагрузке на джобы — **вынести** обработку из веб-процесса (см. §3). | То же; на 2 ядрах отдельный `job`-контейнер часто даёт стабильнее latency веба. |

**Сделано (2026-04-03):** в [`config/deploy.yml`](../config/deploy.yml) задано `RAILS_MAX_THREADS: 5` и комментарии по политике `WEB_CONCURRENCY` для SQLite на 1–2 CPU; в [`config/database.yml`](../config/database.yml) и [`config/puma.rb`](../config/puma.rb) зафиксировано согласование пула с потоками.

**Критерий успеха:** снижение p95 HTML при том же `ab`/реальном трафике, отсутствие роста `SQLite3::BusyException`.

## 2. SQLite на малых CPU (высокий приоритет)

| Мероприятие | Суть |
|-------------|------|
| **Единый `busy_timeout`** | Уже завязан на `SQLITE_BUSY_TIMEOUT_MS` — не занижать на проде при конкурентных записях. |
| **Не смешивать тяжёлые записи** | Пакетные джобы и массовые апдейты — в очереди с **низким параллелизмом**, чтобы не забивать писателей. |
| **Индексы и тяжёлые запросы** | Проверить `EXPLAIN` по горячим путям (каталог, карточка, админка); избегать лишних `COUNT` (частично закрыто кэшем счётчика в каталоге). |
| **Резерв на будущее** | При росте записей и очередей — **PostgreSQL** для primary (отдельный план миграции). |

## 3. Solid Queue и фон (средний–высокий приоритет)

| Мероприятие | Суть |
|-------------|------|
| **`SOLID_QUEUE_IN_PUMA: false`** | В [`config/deploy.yml`](../config/deploy.yml) включить роль **`job`** с `bin/jobs` на том же хосте (второй контейнер) или на отдельной машине — меньше конкуренции за CPU с Puma. |
| **`JOB_CONCURRENCY` и потоки** в [`config/queue.yml`](../config/queue.yml) | На 1–2 CPU держать **мало** процессов/потоков воркера (например 1 процесс, 2–3 потока), увеличивать только по метрикам очереди. |
| **Регулярная уборка** | Уже есть recurring в проекте — следить, чтобы finished jobs не копились без причины. |

## 4. Кэширование и отдача статики (средний приоритет)

| Мероприятие | Суть |
|-------------|------|
| **Solid Cache** | Политика `max_size` / `max_age` в [`config/cache.yml`](../config/cache.yml); следить за ростом `production_cache.sqlite3`. |
| **Фрагментный кэш** | Кэшировать дорогие partial’ы на главной/каталоге, где допустимо устаревание на минуты. |
| **HTTP-кэш** | `Cache-Control`/`ETag` для публичных GET, где нет персонализации (осторожно с сессией и Turbo). |
| **CDN / прокси** | Вынести статику и медиа за Cloudflare или nginx `expires` — разгрузить Ruby. |

---

## 5. Приложение и виды (средний приоритет)

| Мероприятие | Суть |
|-------------|------|
| **Active Storage / варианты** | Не генерировать тяжёлые варианты синхронно в запросе; отдавать уже готовые или lazy через джобы. |
| **N+1** | `includes`/`preload` на списках; проверить карточку товара (галерея, вложения). |
| **Страница товара** | По замерам — длинный хвост latency; профилировать (`rack-mini-profiler`, логи SQL) отдельным шагом. |
| **Сессия** | При росте cookie — рассмотреть `cache_store` для сессий (меньше повторной работы при больших сессиях). |

## 6. Thruster и логирование (низкий–средний приоритет)

| Мероприятие | Суть |
|-------------|------|
| **Thruster** | Уже сжимает ответы — убедиться, что используется в прод-образе Kamal. |
| **`RAILS_LOG_LEVEL=warn`** | Снизить объём I/O логов под нагрузкой. |
| **Healthcheck** | Уже исключён из шума — не дублировать тяжёлые проверки в `/up`. |

## 7. Измерения (обязательный контур)

| Действие | Зачем |
|----------|--------|
| Повторять **`bin/load_test_ab`** / **`bin/load_test_latencies`** после изменений | Сравнение «до/после» на одном стенде. |
| Смотреть **размеры** `storage/*.sqlite3** и очередь Solid Queue | Корреляция с деградацией. |
| При 2 CPU — краткий **профиль CPU** (`perf`, `rbspy` или хотя бы `top`) под нагрузкой | Понять: CPU в Ruby, SQLite, или воркеры. |

## 8. Порядок внедрения (рекомендуемый)

1. Зафиксировать baseline (RPS, p95, размеры БД) в текущей конфигурации Kamal.  
2. Настроить **очередь отдельно от Puma** или снизить `JOB_CONCURRENCY`, если джобы заметны.  
3. Точечно: **фрагментный кэш** + **CDN/статика**.  
4. Профилирование **карточки товара** и тяжёлых SQL.  
5. Пересмотр **потоков Puma** и при необходимости **2 воркера** только при доказанной выгоде на 2 CPU.  
6. Долгосрочно при росте нагрузки на запись — **PostgreSQL**.
